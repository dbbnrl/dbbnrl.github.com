I was so focused on optimizing Haskell last time (both for performance and readability), that I forgot an important issue. I haven’t actually implemented the same algorithm as in the C version:

  • The list of potential factors we’re using in isprime4 to test for primality is the odd numbers less than or equal to sqrt(x), rather than the prime numbers less than or equal to sqrt(x).
  • The use of sqrt is probably quite expensive; the C version squares each factor instead.

Since we’re avoiding sqrt, we can’t simply take odd integers up to a known upper bound, so we have to be a bit more creative. takeWhile will pull from the (infinite) list of odd numbers while a condition is true.

Unfortunately, the runtime jumps way up to 46 seconds. What’s the deal? After some investigation, my conclusion is that fusion is not implemented for takeWhile, although it probably could be. This is unfortunate, but let’s keep going — I expect fusion will also be incompatible with the next optimization. Will it be worth it?

Next, we need to figure out a way to supply a list of prime numbers to isprime, rather than using a list of odd numbers. Where can we find a list of prime numbers?

Version 6 really shows off what you can do with laziness. As our list of prime factors, we supply to isprime6 the infinite list primes6, which is precisely the value we’re trying to compute!

This is a bit circular, and in fact I had to pull the first prime (3) out of the loop and explicitly tack it onto the front of the list to avoid an infinite recursion. After that point, however, isprime6 is guaranteed never to walk farther into primes6 than those values that have already been computed. (To be really operational about it: At any given time, primes6 consists of a finite list of prime numbers followed by an unevaluated thunk which represents the rest of the infinite list. Due to the inRange termination condition, isprime6 can never traverse the list far enough to reach the thunk.)

This version takes 14 seconds. That really isn’t as much of an improvement over version 4 as I’d expect, given the vastly smaller set of factors that are being examined. Now we’re down to 8 seconds — a pretty massive improvement over 14 seconds, and only a factor of 3 or so from the C version’s performance. Although this is a ridiculously simple toy problem, there appears to be room for quite a bit of optimization.

I suspect I’ve taken this about as far as it can go using Haskell’s list datatype. The next step is to investigate array-like alternatives.

At this point, my conclusion is that a highly-readable, compositional coding style is sometimes at odds with performance, because GHC just isn’t quite smart enough 1. I’d be interested in what more experienced Haskell programmers have to say on this subject.

As a side note, observant readers will notice that test in version 7 is a partial function; it will fail on the empty list. It turns out that it actually can’t fail, due to the (f*f) > x termination test — but the compiler has no way of knowing this. I wonder what the implications of this are from an optimization standpoint?

Another operator section, used with the any function, which is just short-circuit boolean OR over a list. Version 4 is nice and clean, and much more pleasant to read than the C version. I particularly like how isPrime4 reads almost like an English sentence: “X is prime if it is not the case that any elements equal zero in the list produced by…”

This version completes in 83 seconds. So… why was version 1 so much faster?

Version 4 is structured as a pipeline. Each stage consumes a list, performs an operation on each element, and produces a new list (or a single result, in the case of any). In a traditional non-lazy language, this code style would clearly be wasteful: First generate in memory a list of odd integers up to sqrt(x), then traverse that list and generate a new list with the (x `rem`) operation applied, then traverse that list looking for elements equal to zero.

In Haskell, it doesn’t really work that way due to lazy evaluation. Each list is computed only as it is needed, so the various operations proceed in lockstep (they are essentially coroutines). But there’s still a lot of overhead involved in allocating list elements, filling them, pulling items out of them, and freeing them.

Ideally, we’d like the compiler to be smart enough to transform version 4 into something like version 1. Why isn’t it?

Well, it is, if we add the following line above primes4:

With this type annotation, version 4 now completes in 17 seconds. Why the difference? Well, since we didn’t originally provide a type for primes4, ghc was free to infer any reasonable type 1. And it chose Integer, which is an arbitrary-sized integer. We really want to use something like Int, which uses the (bounded) native machine representation. With a type annotation declaring that we want a list of Int, the behavior changes dramatically.

Let’s add primesX :: [Int] type annotations to every version and compare:

  • Version 1 with [Int]: 24 seconds
  • Version 2 with [Int]: 32 seconds
  • Version 3 with [Int]: 40 seconds
  • Version 4 with [Int]: 17 seconds

As you can see, the time increases until we reach version 4, which is written entirely in terms of standard list processing functions (no test helper function). The compiler can now apply short cut fusion. The result is now faster than the “manually” created implementation in version 1!

I was actually slightly surprised that ghc wouldn’t perform fusion for Integer lists. Perhaps the optimization is only implemented for unboxable types. Or, perhaps I’m completely wrong about what’s going on here. At some point, I should revisit this analysis using a tool like ghc-core.

Haskell is down to 17 seconds. Making progress!

  1. This is misleading, perhaps. Normally, I believe Haskell will try to infer the most general type. In this case, that would be something like Integral a => [a], which means a list of any a such that a belongs to the Integral typeclass. However, inferring that type could result in primes4 being reevaluated each time it is referenced, due to the hidden typeclass parameter. This could be surprising behavior, given that syntactically, primes4 looks like a constant, not a function. Haskell’s monomorphism restriction was designed to prevent programmer confusion in these cases by forcing the compiler to infer a monomorphic (non-polymorphic) type. Since any member of the Integral typeclass is an equally resonable choice for monomorphic type in this case, Haskell consults an explicit list of defaults to arrive at the choice of Integer.↩

I’ve been reading a ton of Haskell papers, blogs, and other resources. I’ve also made small inroads into doing OpenGL rendering in Haskell, but it’s become apparent that I should back off and tackle smaller mountains before I go there.

It occurred to me the other day that I should revisit an old memory. Back when I first taught myself to program (BASIC, on my old Apple IIe), I spent quite a bit of time optimizing a program to generate prime numbers.

This seems like a nice starting point for diving into Haskell. The initial implementation will likely be very trivial, but in the course of optimization I’m hoping this can lead to a beginner’s understanding of:

  • Utilizing laziness
  • Dealing with the consequences of laziness
  • Writing idiomatic code
  • Benchmarking, profiling, and analyzing generated code (maybe learn to read GHC-core?)
  • Array/Vector libraries (assuming linked lists don’t scale well to this problem)
  • Using imperative (mutable) Array/Vector operations

The algorithm I’ll be implementing and optimizing uses trial division to enumerate the primes. This is not necessarily the best way to find primes, but that isn’t the point here — I’m trying to optimize the implementation, not find the best algorithm. Maybe I’ll revisit that question later.

A Bit Late to the Party

Jan 9th, 2013 | Comments

So, I’ve finally decided to try out this blogging thing.

Recently I’ve been fiddling around on a project that (so far) involves simultaneously learning modern OpenGL and Haskell. And Functional Reactive Programming (FRP). Oh, and a bit of WXWindows, possibly.

All of which involves getting a lot of concepts straight in my head, which I’m thinking might be easiest if I take notes, some of which might actually be organized enough for other people to find useful, which suggests that I should look into blogging…

In my typical premature-optimization, tool-focused style, I created this blog on wordpress.com, discovered through further research that it wasn’t quite perfect for my needs, and abandoned it the same day (after zero posts) to start getting octopress-over-github configured. This way is much geekier, which makes it Better.

Oh, and I keep meaning to buy some parts and do a bit of Arduino-based robotics, though at this point I’ve spent so much time thinking about doing that without actually doing it (including hours of research and a carefully constructed shopping list) that Raspberry Pi may now be a better option. I should really research that…

There’s a pattern here, which I hope to break out of by:

  • Doing things in small increments and committing code to Git frequently, and
  • Blogging frequently about what I’m working on.

The former has helped my productivity at work quite a bit. (And helped curtail my tendency toward over-design and premature optimization). The latter is just the application of the same principle to mental output that isn’t code and doesn’t go into revision control (though with Github pages, there’s a certain additional symmetry…)

I’m not quite sure why I’m telling myself these things, it’s not as if the Internet is listening. I suppose my audience is really Future Dan.

Hello, Future Dan.
